# Chat 功能异常修复计划

您在日志中遇到了 `net::ERR_ABORTED` 错误，这通常意味着请求被意外中断或拒绝。结合之前的上下文，我发现了一些关键问题：

## 根本原因分析

1.  **URL 编码问题**：
    *   请求 URL 中包含中文字符 `username=%E7%8E%8B%E7%A8%8B%E8%90%A7`（王程萧）。虽然现代浏览器通常能处理，但在某些代理或后端配置下，未正确解码的参数可能导致 400/422 错误或路由失败。

2.  **API 路径不一致**：
    *   虽然 `vite.config.js` 配置了 `/api` 代理，但 `useLLMStream.js` 和 `Chat.vue` 中的 `baseUrl` 默认为空字符串。这在开发环境（`localhost:3000`）通常没问题，但为了稳健性，最好明确使用 `/api` 前缀。

3.  **流式请求中止**：
    *   `ERR_ABORTED` 最常见的原因是前端代码主动调用了 `AbortController.abort()`，或者请求超时。
    *   在 `Chat.vue` 中，每次发送新消息时都会创建一个新的 `AbortController`。如果之前的请求未完成就被新的操作打断，或者组件卸载时触发了 `abort()`，就会报这个错。

## 修复步骤

### 1. 完善 URL 参数编码
确保所有通过 URL 传递的参数（如 `username`）都经过 `encodeURIComponent` 处理。

### 2. 增强 `fetch` 调用的错误处理
在 `Chat.vue` 和 `useLLMStream.js` 中，明确区分 `AbortError` 和其他网络错误，避免将用户主动取消的操作视为系统故障。

### 3. 统一 API 路径
确保所有请求都正确指向 `/api/...`。

### 4. 优化 `AbortController` 生命周期
确保仅在真正需要取消请求（如用户点击停止、组件销毁）时才调用 `abort()`，避免意外中断。

我将重点检查 `Chat.vue` 中的 `fetchSessions` 和 `sendMessageStream` 方法，确保它们正确处理了上述逻辑。
